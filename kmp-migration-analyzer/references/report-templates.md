# Report Templates

The skill writes Markdown into the analyzed project's `docs/migration/` folder. Two report types:
the whole-project **roadmap** (Stage 1) and a per-batch **deep report** (Stage 2). Both are
**sequential** — read top to bottom and worked through in order. Follow these skeletons; scale each
section to what the project actually contains (omit a section only if it genuinely doesn't apply, and
say so rather than leaving it blank).

Every concrete finding cites `file_path:line`. Every safe stopping point uses a `✅ CHECKPOINT`
marker with its verification command (see `checkpoints.md`).

---

## `00-roadmap.md` (Stage 1 — Survey)

```markdown
# KMP Migration Roadmap — <Project Name>

> Generated by kmp-migration-analyzer. This is an ANALYSIS, not a migration. The plan is a
> behavior-identical, 1:1 translation: nothing about how the app works should change. Work through
> it top to bottom; stop and verify at every ✅ CHECKPOINT.

## Executive summary
- App shape: <single-module | multi-module per feature/layer>, <N> modules.
- Overall readiness: <one-line verdict>.
- Biggest blockers: <e.g. Views (no Compose), Groovy Gradle, Retrofit, RxJava>.
- Proposed approach: <N> batches; first checkpoint reachable after <…>.

## Module graph
<list or simple tree of modules with their dependency edges; mark pure-Kotlin/JVM modules that must
become explicit KMP modules>

## Readiness heat-map
| Module | Status | Blockers (Phase A) | Notes |
|---|---|---|---|
| :core:domain | 🟢 KMP-clean | — | pure Kotlin → must become explicit KMP module |
| :feature:auth:data | 🔴 blockers | Retrofit, Moshi | … |
| :app | 🔴 blockers | Groovy, Views, Hilt | … |
<🟢 clean / 🟡 minor / 🔴 significant>

## Global migration order

3. Phase-A library swaps (lowest layer first): <ordered list>

<reference ordering-rules.md reasoning briefly>

## Proposed batches
- **Batch 01 — <name>**: <modules>. Ends at: ✅ CHECKPOINT — <what runs>.
- **Batch 02 — <name>**: <modules>. Ends at: ✅ CHECKPOINT — <what runs>.
- …
<each batch sized to one working session; topologically ordered>

## Checkpoint plan
<the ordered list of checkpoints across the whole migration, each with its verification command>

<NDK? alpha versions? unknown libs? — each with what was found, options, and a recommendation.
```

## Single combined report (small / single-module project)

When the project is one module (or small enough for one pass), don't split: produce a single
`00-roadmap.md` that includes the roadmap sections **and** the deep-analysis sections inline (one
"batch" = the whole app), keeping the same ordered-task-list-with-checkpoints structure.
